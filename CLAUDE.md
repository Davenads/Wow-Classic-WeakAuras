# CLAUDE.md — WoW Classic WeakAuras Workshop

This repo authors and version-controls **WeakAuras for World of Warcraft Classic**.
This file is the operating guide: read it before creating or editing any aura, and
follow the conventions here exactly. It links to deeper references in `docs/` — open
those when you need exact field names or API details rather than reproducing them here.

> **Target platform: WoW *Classic*, not Retail.** Custom Lua runs on a Classic client.
> Retail-only APIs are absent and many spell IDs differ. When in doubt, consult
> `docs/classic-api.md` before writing Lua — do **not** assume Retail behavior.

---

## 1. What "authoring a WeakAura" means here

A WeakAura's portable form is a single opaque `!WA:2!...` **export string** (a serialized,
compressed, base64 Lua table). You cannot meaningfully hand-write that string. So the
real authoring loop is:

1. Build/skeleton the aura **in-game** (regions, triggers, layout) — the WA UI owns geometry.
2. Author the **custom Lua** (triggers, conditions, text, actions) as `.lua` files **here**,
   where it can be linted, diffed, and reviewed. Paste it into the in-game **Expand** editor.
3. Export the finished aura and save the string as `export.txt` — the **source of truth**
   for reimporting the whole aura.
4. Record metadata + changelog in `aura.md`.

Optionally, decode the whole string to an editable table with `tools/` for programmatic
edits or full-aura diffs (see `docs/workflow.md` §2). Default to the code-file workflow;
reach for full decode only when you must.

**Never claim an aura "works" unless it has been imported and tested in-game.** This repo
cannot run WoW. Custom Lua can be linted (`luacheck`) but not executed here.

## 2. Directory structure

```
Wow-Classic-WeakAuras/
├── CLAUDE.md                 # ← this file: operating guide + conventions
├── README.md                 # human-facing overview + catalog
├── .luacheckrc               # luacheck config (WoW + aura_env globals pre-declared)
├── .editorconfig             # 4-space Lua, LF endings
├── .vscode/settings.json     # LuaLS + Ketho wow-api setup
├── auras/                    # ONE folder per aura (or aura group)
│   ├── _TEMPLATE/            # copy this to start a new aura — do NOT edit in place
│   │   ├── aura.md           #   metadata, wago link, purpose, changelog
│   │   ├── export.txt        #   the !WA:2! string (source of truth)
│   │   ├── aura.json         #   OPTIONAL decoded table (generated by tools/decode.js)
│   │   └── code/             #   custom Lua, one file per code block (see §5)
│   │       ├── trigger.lua
│   │       ├── untrigger.lua
│   │       ├── tsu.lua
│   │       ├── conditions.lua
│   │       ├── custom_text.lua
│   │       ├── init.lua
│   │       ├── on_show.lua
│   │       └── on_hide.lua
│   └── <class-or-category>/<AuraName>/   # real auras live here
├── shared/                   # reusable Lua snippets shared across auras
├── lib/                      # stub .toc so vscode-wow-api activates (autocomplete)
├── tools/                    # Node scripts: decode/encode !WA:2! strings
└── docs/
    ├── weakauras-reference.md  # data model, triggers, TSU, conditions, string format
    ├── classic-api.md          # Classic flavors, API differences, version numbers
    └── workflow.md             # round-trip tooling, aura_env sandbox, editor setup
```

**Organizing auras:** group by class then aura (`auras/Mage/FrostboltProc/`), or by
category for non-class auras (`auras/Raid/DispelTracker/`, `auras/UI/CastBar/`). One
folder = one importable unit (a single aura, or a group + its children exported together).

## 3. Creating a new aura — checklist

1. **Copy the template:** duplicate `auras/_TEMPLATE/` to `auras/<Category>/<AuraName>/`.
   Delete any `code/*.lua` files the aura doesn't use — keep only the hooks it actually has.
2. **Fill `aura.md`** — name, target flavor(s), purpose, wago URL (if any), and start the
   changelog. Convert relative dates to absolute (today is knowable from the environment).
3. **Write custom Lua** in `code/` against `docs/weakauras-reference.md` §3–4 and
   `docs/classic-api.md`. Lint with `luacheck` before considering it done.
4. **Paste into the game**, wire up triggers/conditions/regions, test.
5. **Export** and paste the string into `export.txt` (single line, no trailing newline).
6. Optionally run `tools/decode.js` to write `aura.json` for a diffable view.
7. Update the catalog in `README.md`.

When editing an existing aura, update **both** the `code/` files **and** re-export
`export.txt` (the string is what actually gets imported — stale strings are the main
failure mode). Bump the changelog in `aura.md`.

## 4. Classic API rules (summary — full detail in `docs/classic-api.md`)

- **Know the flavor.** Classic Era / SoD / Hardcore share one client (old talents, **no
  `GetSpecialization`**). Cata Classic and MoP Classic are separate clients that **do**
  have the spec system. Branch on `WOW_PROJECT_ID` when code must span flavors.
- **`UnitAura` / `UnitBuff` / `UnitDebuff` exist on Classic** (removed on Retail). Use them.
  `C_UnitAuras` is backported to current clients but `UnitAura` is the safest baseline.
- **`GetSpellInfo` / `GetSpellCooldown` exist on Classic** (removed on Retail). Prefer the
  legacy globals for compatibility, or branch.
- **Spell ranks are separate spell IDs** in Classic. Tracking a ranked spell means tracking
  **all** rank IDs or matching by **name**. This is the #1 "my aura doesn't fire" bug.
- **Combat log:** register `COMBAT_LOG_EVENT_UNFILTERED`, read via
  `CombatLogGetCurrentEventInfo()`, and **filter by subevent** (`CLEU:SPELL_AURA_APPLIED`).
- **Guard Retail-only namespaces** (`C_ClassTalents`, `C_Traits`, `GetSpecialization` on
  Era) with `if C_Foo then` — they're nil on Classic.
- **Look up IDs on the flavor-specific Wowhead** (`wowhead.com/classic/`, `/cata/`), not
  the Retail DB. SoD rune spells have their own IDs.

## 5. Custom-code conventions

Custom Lua runs in a per-aura **sandbox** with `aura_env` as context. Full reference:
`docs/workflow.md` §6. Quick facts:
- `aura_env.id`, `aura_env.uid`, `aura_env.region`, `aura_env.state`, `aura_env.states[n]`,
  `aura_env.config.<optionKey>` (Custom Options values), `aura_env.saved` (persistent).
- Blocked: `getfenv`/`setfenv`/`loadstring`/`pcall`, `SendMail`, `CreateMacro`, and
  sensitive tables. Don't write code that depends on them.

**One file per code block** in `code/`, matching where it pastes in the WA UI:

| File | WA code block | Signature |
|---|---|---|
| `trigger.lua` | Trigger → Custom (event/status) | `function(event, ...)` → bool |
| `tsu.lua` | Trigger → Custom (Trigger State Updater) | `function(allstates, event, ...)` → bool |
| `untrigger.lua` | Trigger → Custom Untrigger | `function(event, ...)` → bool |
| `conditions.lua` | Conditions → Custom Check / Custom Code | `function() return bool end` etc. |
| `custom_text.lua` | Display → text `%c` functions | `function() return "..." end` (one per `%c`) |
| `init.lua` | Actions → On Init | setup, define helpers on `aura_env` |
| `on_show.lua` | Actions → On Show | runs when the aura shows |
| `on_hide.lua` | Actions → On Hide | runs when the aura hides |

Conventions:
- Start each file with a one-line comment naming the aura and the exact WA block it pastes into.
- Prefer numeric spell IDs over names; put magic IDs in named locals at the top with a
  comment linking the Wowhead page.
- Keep each block self-contained — WA blocks don't share scope except through `aura_env`.
- Every `.lua` file must pass `luacheck` clean. Reusable logic goes in `shared/`.

See `shared/` for reusable snippets and `auras/_TEMPLATE/code/` for annotated stubs.

## 7. Project conventions

- **Don't invent spell IDs, API signatures, or WA field names.** Verify against `docs/`
  or the linked Wowhead/wiki pages. If unsure, say so rather than guessing — a wrong spell
  ID silently breaks an aura with no error.
- **Keep `export.txt` and `code/` in sync.** The string is authoritative for import; the
  files are authoritative for review. A change to one demands a change to the other.
- **Test in-game before declaring done.** Linting ≠ working.
- Follow the global commit conventions (concise, imperative, no attribution). This repo is
  not yet a git repo — run `git init` before the first commit if the user wants history.
